# AiCommander A/B/C 升级路线图设计

> 生成日期：2026-06-05
> 状态：路线图已确认
> 推荐顺序：A 工作流闭环 -> B AI 研判包 -> C 大屏运营展示

---

## 背景

AiCommander 当前定位是“涉油案件数智研判与防控辅助系统”。系统应服务案件录入、字段质量检查、模式识别、时空规律、链条线索、热点区域、报告生成和部署建议，不扩展为巡逻执行平台、红色网格流程平台、警企联动闭环平台或通用综合管理平台。

上一轮已经形成三条升级方向：

- A：录入预检到待办闭环，提升日常使用便利性。
- B：AI 研判包与报告自动化，提升研判材料产出效率。
- C：大屏运营展示，把 A/B 的结果呈现给领导和现场指挥。

本设计将三条路线合并为一个分阶段路线图。核心判断是先把后台工作流跑顺，再让 AI 产出有可复核依据，最后把可信结果上大屏。

---

## 总体目标

本轮升级不追求新增更多页面，而是把现有能力串成闭环：

```
案件录入/导入
  -> 保存前预检
  -> 后台批处理
  -> 待办中心分流
  -> 研判包/经验卡/结论草稿
  -> 人工确认
  -> 报告与大屏引用
```

成功后，用户不需要逐个页面寻找缺项，也不需要手动判断下一步应该去哪里。系统自动把“缺什么、影响什么、下一步做什么”整理成可执行待办；AI 只生成可核查的事实、推断和建议草案；大屏只展示趋势、缺口和复核状态。

---

## 路线选择

### 推荐路线：A -> B -> C

先做 A，因为录入预检、批处理和待办中心是后续 AI 与大屏的真实数据来源。没有 A，B 会缺少稳定上下文，C 会出现空指标或口径不清。

### 展示优先路线：C -> A -> B

仅适合近期必须做汇报演示。风险是大屏可能只能展示静态空态，不能证明日常便利性提升。

### AI 优先路线：B -> A -> C

仅适合近期急需研判报告材料。风险是 AI 产出缺少流程承接，容易变成一次性文本生成，而不是系统能力。

本轮采用推荐路线。

## Phase 0：基线固化

### 目标

确认当前项目真实状态，固定升级边界和验收入口。Phase 0 不新增业务能力，只为 A/B/C 建立可回归基线。

### 范围

- 确认核心页面入口：
  - `/cases`
  - `/suggestions`
  - `/case-intelligence`
  - `/intelli-inspect`
  - `/reports`
  - `/conclusions`
  - `/dashboard`
- 确认核心后端入口：
  - `POST /api/cases/structure-preview`
  - `POST /api/cases/preprocess/batch`
  - `POST /api/cases/batch-review`
  - `GET /api/suggestions/`
  - 案件研判工作台与模型上下文包接口
  - 数智告警研判包接口
- 记录当前工作区已有未提交改动，后续实现只改与阶段目标有关的文件。

### 验收

- 后端关键测试可定向运行。
- 前端关键模型测试可定向运行。
- 本地前后端能启动并打开核心页面。

## Phase A：录入预检到待办闭环

### 业务目标

让案件从录入或导入开始就自动进入“质量检查 -> 后台处理 -> 待办分流”的闭环。用户只需要处理系统给出的下一步，不需要自己判断缺项影响哪些研判模块。

### A1 保存前预检

案件录入页在保存前展示对后续能力的影响：

- 地图研判：是否有地点和合法坐标。
- AI 预处理：案情描述长度和关键事实是否足够。
- 结论分层：是否具备事实、推断、建议的最低输入。
- 经验卡：是否具备作案条件、发现方式、防护短板和证据缺口。
- 奖金核算：只提示影响金额的指标和材料缺口，不在普通研判页展开内部核算。

预检只提示风险，不阻止案件保存。阻断只用于明确会导致错误计算或错误提交的场景。

### A2 后台批处理

批处理统一执行以下动作：

- 对未处理案件执行预处理。
- 刷新案件质量评分。
- 生成或更新经验卡草稿。
- 检查报告摘要依据是否充分。
- 检查奖金指标和佐证材料门禁。
- 输出 `job_id/status/progress/processed/failed/skipped/issues`。

第一期可以沿用现有批处理接口形态；后续如需要跨服务重启保留进度，再引入持久化任务表。

### A3 待办中心分流

`/suggestions` 继续作为统一待办入口，分类保持可解释：

- `data_quality`：坐标、案情、关键字段缺口。
- `analysis`：预处理、研判上下文、相似条件分析。
- `review`：结论分层、链条推断、报告结论复核。
- `workflow`：事件转案件、会议报告沉淀。
- `bonus`：奖金指标与材料缺口，入口只跳案件/奖金页。
- `alert`：数智告警研判包核查。
- `experience`：经验卡生成与人工确认。
- `report_quality`：报告依据不足、会议结论未沉淀。

`GET /api/suggestions/` 必须保持读接口语义，不应因生成待办而提交数据库修改、覆盖质量分或更新经验卡时间。

### A4 操作入口

待办项的按钮只做三类动作：

- 直接处理：如单案预处理、事件转案件。
- 跳转复核：如结论详情、经验卡、奖金核算、告警研判包。
- 提示人工处理：对不适合自动执行的待办给出边界说明。

### A5 A 阶段验收

- 新案件保存后，能在案件页看到影响后续研判的预检信息。
- 批处理完成后，能形成明确 issues，并进入待办中心。
- 待办中心每类待办都有明确目标页或处理动作。
- 读接口不产生写库副作用。

## Phase B：AI 研判包与报告自动化

### 业务目标

让 AI 介入非结构化材料整理和研判表达，但不替代规则评分、事实确认、处置决策和人工复核。

### B1 统一 AI 输出结构

AI 产出统一分为：

- `facts`：来自案件字段、案情、证据、数智告警或研判工作台的事实。
- `inferences`：基于事实的模式推断，必须带依据和置信度。
- `recommendations`：防控参考或补齐建议，不能写成已执行任务。
- `information_gaps`：影响置信度和报告质量的信息缺口。
- `evidence_refs`：可追溯来源，如案件编号、字段、标签、相似条件、告警编号。
- `boundary`：不预测犯罪、不自动派发、不确认串案、不替代人工审核。

### B2 模型上下文包

案件研判工作台负责提供模型上下文，AI 不直接绕过规则服务查询或评分。上下文包包括：

- 选中案件基础信息。
- 标签与依据。
- 相似条件案件。
- 时空规律和区域画像。
- 经验卡草稿。
- 信息缺口。
- 系统边界说明。

### B3 研判包与报告

系统生成三类可复制材料：

- 案件研判包：用于单案复盘、经验沉淀和同类案件对比。
- 数智告警研判包：用于告警事实、AI 依据、信息缺口和下一步核查建议。
- 报告/结论草稿：用于会议报告、情报结论和领导汇报前的人工复核。

所有材料都应支持 Markdown 复制，并显示事实、推断、建议和缺口。

### B4 AI 降级与错误处理

- 未配置模型时，使用确定性规则输出基础研判包。
- 模型失败时，保留错误提示并继续输出规则侧结果。
- JSON 解析失败时，降级为结构化兜底，不把原始模型文本直接写入正式结论。
- 低置信度内容必须进入人工复核，不得自动发布。

### B5 B 阶段验收

- 任意 AI 研判包都能追溯事实来源。
- AI 建议不会显示为已执行任务。
- 模型不可用时，系统仍能输出可用的规则侧材料。
- 结论和报告入口能区分草稿、待复核、已确认。

## Phase C：大屏运营展示

### 业务目标

把 A/B 的结果展示给领导和现场指挥：趋势、重点区域、链条待确认、AI 产出、数据缺口、待复核事项。大屏是展示和入口，不是执行台。

### C1 展示内容

大屏展示以下信息：

- 本月或近 30 天案件趋势。
- 坐标完整率和关键字段完整性。
- 数智告警待核查数量。
- 经验卡、报告、结论草稿产出状态。
- 链条推断待确认数量。
- 待办中心高优先级事项。
- 系统产出质量空态或可解释指标。

### C2 口径要求

所有数字必须标明口径：

- 本月。
- 近 30 天。
- 近 7 周。
- 全量。

不同口径不能放在同一个指标里混用。无法计算的指标显示解释性空态，不用固定假数字。

### C3 交互边界

大屏只允许：

- 查看趋势。
- 查看待复核摘要。
- 跳转到对应业务页面。
- 展示地图点位、热区和链条关系口径。

大屏不允许：

- 确认或驳回链条推断。
- 提交奖金核算。
- 派发巡逻或外勤任务。
- 把未确认推断写成已查实事实。

### C4 C 阶段验收

- `/dashboard` 在 1280x720 首屏内稳定展示主要信息。
- 地图、趋势、待办和 AI 产出均有数据来源或明确空态。
- 顶部指标口径明确，不出现“本月”和“全量”混淆。
- 大屏所有操作都只跳转，不在大屏直接提交高风险业务动作。

## 非目标

本轮明确不做：

- 自动巡逻派发。
- 犯罪预测或未来地点预测。
- 自动确认串案或链条事实。
- 自动审批、自动发布结论。
- 红色网格流程平台。
- 警企联动闭环平台。
- 通用综合管理平台。
- 在大屏展示或提交奖金核算结果。
- 公共地图自动补全专项。

公共地图自动补全、外部真实设备接入和持久化任务队列可以作为后续专项，不并入本轮 A/B/C 主线。

## 风险与控制

| 风险 | 控制方式 |
| --- | --- |
| 待办生成顺手写库，造成读接口副作用 | `/suggestions` 只读；需要更新的动作放到显式 POST/PATCH 接口 |
| AI 把推断写成事实 | 统一 schema 分离 facts/inferences/recommendations，并保留 evidence_refs |
| 大屏指标混用口径 | 每个指标必须写明本月、近 30 天、全量或近 7 周 |
| 奖金核算侵入研判主线 | 奖金只进入案件/奖金页，不上大屏，不参与研判报告主口径 |
| 路径复盘被误解为预测 | 文案固定为“已发生案件路径条件复盘”，不输出未来地点 |
| 页面新增过多导致使用复杂 | 优先复用现有页面和入口，不新增综合管理大页 |

## 实施切分建议

### 第一轮：A 阶段最小闭环

交付录入预检、批处理结果展示、待办中心跳转和读接口副作用修正。该轮完成后，系统能自动告诉用户下一步该做什么。

### 第二轮：B 阶段研判包

交付统一 AI 输出结构、Markdown 复制、结论草稿复核和模型失败降级。该轮完成后，AI 产出能进入人工复核与报告链路。

### 第三轮：C 阶段大屏运营

交付大屏指标口径、待办摘要、AI 产出状态和数据缺口展示。该轮完成后，领导视图能展示真实系统工作成果。

---

## 最终验收

- 从案件录入到待办分流形成闭环。
- 从研判包生成到人工复核形成闭环。
- 从后台产出到大屏展示形成闭环。
- 所有 AI 内容都有事实来源、推断边界和人工复核入口。
- 所有大屏指标都有口径或解释性空态。
- 项目边界仍保持“涉油案件数智研判与防控辅助系统”。
